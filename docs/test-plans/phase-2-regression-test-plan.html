Phase 2 Regression Test Plan

Baseline-protection plan for read-only integration boundaries and Phase 0/1 carry-forward constraints.

Versionv0.1
StatusDraft baseline

Protected Baselines

  • Coinbase Advanced remains the only exchange surface.
  • Read-only mode remains enforced throughout Phase 2.
  • Spot-only and trade-only scope remain intact.
  • USD remains the locked valuation currency.
  • Approved candidate class and DOGE watchlist treatment remain visible and intact.
  • No unsupported funding, transfer, staking, Web3, margin, or derivatives surfaces are introduced.

Execution Environment and Evidence

  • Run regression review whenever Coinbase integration behavior, credential rules, refresh behavior, capital interpretation, or protected policy surfaces change materially.
  • Primary review inputs: approved roadmap, approved Phase 2 PRD, approved Phase 2 TRD, approved Phase 2 test plans, Phase 0 decision memo, and the latest implementation/test-result artifacts.
  • Expected evidence: comparison checklist, identified drift items if any, approval notes for intentional changes, and a durable result artifact under docs/test-results/phase-2-regression-results.md or equivalent.

Required Regression Checks

  • RTP2-01: No write-capable Coinbase endpoints or operator actions are introduced.
  • RTP2-02: Account truth does not silently override approved policy overlays.
  • RTP2-03: Read-only failures do not crash the shell or remove the inactive-trading boundary.
  • RTP2-04: Existing-funds-only interpretation remains intact.
  • RTP2-05: Phase 1 update-check, config, secret-handling, and ignored-file guarantees remain intact.
  • RTP2-06: Startup refresh remains a one-time safe startup behavior and does not drift into silent polling.

Execution Contract

RTP2-01: No Write-Capable Coinbase Path

Verify that no Phase 2 service, command, or UI action introduces order placement, order cancellation, or any other write-capable Coinbase path.

Automation: Required.

Review method: codepath review, command registration review, and UI affordance inspection.

Expected evidence: reviewed navigation and command map with no write-capable exchange action present.

RTP2-02: Local Policy Overlay Preservation

Verify that live Coinbase refresh cannot silently overwrite approved candidates, DOGE watchlist treatment, or protected-asset policy overlays.

Automation: Required.

Review method: compare pre-refresh and post-refresh policy state plus codepath review for overlay merges.

Expected evidence: overlay comparison artifact showing no drift.

RTP2-03: Safe Failure Retention

Verify that read-only failures do not crash the shell, remove inactive-trading language, or degrade Phase 1 safety boundaries.

Automation: Required.

Review method: forced-failure runs for missing credentials, rejected auth, and transport failure.

Expected evidence: failure-state checklist showing retained shell safety boundaries.

RTP2-04: Existing-Funds-Only Protection

Verify that no new shell surface implies off-platform funding, external transfer, or unsupported capital sources.

Automation: Required.

Review method: navigation review against approved scope plus account-summary inspection.

Expected evidence: screen checklist showing absence of unsupported funding surfaces.

RTP2-05: Phase 1 Baseline Preservation

Verify that Phase 1 update-check, config, secret-handling, and ignored-file guarantees remain intact after Coinbase read-only integration is introduced.

Automation: Required.

Review method: compare the Phase 2 implementation and results against the last approved Phase 1 baseline artifacts.

Expected evidence: regression comparison record with no weakened Phase 1 guarantees.

RTP2-06: Startup Refresh Boundary Preservation

Verify that the approved Phase 2 startup refresh behavior remains a one-time safe startup action and does not drift into silent background polling or retry loops without explicit approval.

Automation: Required.

Review method: compare current startup behavior against the approved refresh contract and inspect for recurring background refresh paths.

Expected evidence: startup-behavior review artifact showing one-time startup refresh only.

Execution Method

  • Run regression review whenever the Phase 2 baseline or any protected boundary changes materially.
  • Compare new revisions against the approved roadmap, Phase 0 carry-forward decisions, approved Phase 2 PRD, and approved Phase 2 TRD.
  • Where implementation exists, compare current behavior and validation artifacts against the last approved result set rather than relying on document review alone.
  • Reject any silent scope expansion, weakened read-only boundary, or loosened safety contract that lacks explicit approval.

Pass Condition

Phase 2 adds truthful exchange visibility without weakening the approved product boundary or the safe foundation established in Phase 1.
