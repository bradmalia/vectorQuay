Regression Test Plan

Phase 3 Regression Test Plan

Baseline-protection plan for persistence without execution or secret leakage.

Execution Environment and Evidence

  • Run regression review whenever persistence scope, recovery behavior, audit/event schema, or retained-history rules change materially.
  • Primary review inputs: approved roadmap, approved Phase 2 closeout artifacts, approved Phase 3 PRD/TRD, and the latest implementation/test-result artifacts.
  • Expected evidence: comparison checklist, identified drift items if any, and a durable result artifact under docs/test-results/phase-3-regression-results.md or equivalent.

Protected Baselines

  • Coinbase Advanced remains the only exchange surface.
  • Read-only mode remains enforced throughout Phase 3.
  • No execution, no cancellation, and no paper-trading behavior is introduced.
  • Secrets remain external and must not enter durable state or audit payloads.
  • Startup/manual refresh remains explicit; persistence must not drift into silent polling.

Required Regression Checks

  • RTP3-01: persistence introduces no write-capable Coinbase path.
  • RTP3-02: persisted payloads exclude secrets and secret-derived plaintext.
  • RTP3-03: startup recovery does not label restored state as fresh live state incorrectly.
  • RTP3-04: local policy overlays remain authoritative after recovery and subsequent refresh.
  • RTP3-05: failure/corruption fallback retains safe shell behavior.
  • RTP3-06: Phase 2 no-polling and inactive-trading boundaries remain intact.
  • RTP3-07: when no durable state exists, Phase 2 empty/non-connected behavior remains intact and no false recovered-state cues appear.

Execution Contract

RTP3-01: No Write-Capable Drift

Automation: Required.

Review method: codepath review plus integration verification that persistence adds no order-placement or cancellation affordances.

Expected evidence: command map / service-path review showing no write-capable exchange action present.

RTP3-02: Secret Exclusion Preservation

Automation: Required.

Review method: persisted-payload inspection and automated serialization tests.

Expected evidence: sample durable payloads and passing automated assertions showing no secret leakage.

RTP3-03: Restored-State Truthfulness

Automation: Required.

Review method: startup recovery test with failed/absent refresh follow-up.

Expected evidence: shell-state record showing restored data is labeled distinctly from live data.

RTP3-06: No-Polling / Inactive-Trading Preservation

Automation: Required.

Review method: compare current startup and refresh behavior against the approved Phase 2/3 contract.

Expected evidence: review artifact showing one-time startup recovery plus explicit live refresh behavior only.

RTP3-07: No-Recovery-Data Baseline Preservation

Automation: Required.

Review method: run the shell with no persisted state present and compare behavior against the approved Phase 2 never-loaded / non-connected baseline.

Expected evidence: startup-state record showing no false recovered-state label and no degraded empty-state behavior.

Pass Condition

Phase 3 adds durable local state and auditability without weakening the approved read-only, no-execution, and no-secret-leakage boundaries established in earlier phases.
