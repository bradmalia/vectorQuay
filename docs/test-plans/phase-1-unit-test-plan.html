Unit Test Plan

Phase 1 Unit Test Plan

Component-level validation for configuration, shell state, placeholder views, and safety boundaries in the Phase 1 baseline.

Versionv1.0
StatusApproved
OwnerBrad Malia
ThemeIndigo Signal

Purpose

Phase 1 unit testing validates the smallest technical contracts in the approved shell baseline: configuration precedence, local persistence, redaction behavior, view-model state, and reserved-feature guardrails.

Test Goal

Prove that individual Phase 1 components behave deterministically and safely before broader integration testing begins.

Validation Areas

  • Configuration precedence and effective-value resolution.
  • Local settings serialization and persistence.
  • Secret-source detection and redaction.
  • Shell navigation and top-level view state.
  • Placeholder-state rendering contracts.
  • Policy-edit and threshold-profile state transitions.
  • Source-registry state handling.
  • Manual update-check result mapping.
  • Unsupported account-surface exclusion.

Execution Environment and Fixtures

  • Primary execution environment: Ubuntu-first local development environment with the Phase 1 .NET 8 solution and test projects present.
  • Expected test command baseline: dotnet test executed from the repository root or solution root.
  • Until the exact test project names are finalized in implementation planning, the command forms in this document should be read as patterns using the eventual core-test and app-test project paths under tests/.
  • Required fixture set: deterministic application defaults fixture, checked-in template fixture, temporary settings.json fixture, temporary secrets.env fixture, environment-variable override fixture, release-source response fixtures for update-check mapping, and log-capture fixture for redaction assertions.
  • Test runs must isolate local settings and secret fixtures from the real operator paths by using temporary directories and process-scoped environment-variable overrides.
  • Expected evidence: unit-test summary output, failing-test details if any, and a durable result record under docs/test-results/phase-1-unit-results.md or equivalent.

Unit-Level Checks

UTP1-01: Configuration Precedence Resolution

Verify that defaults, checked-in templates, application-managed local settings, external secret files, and environment-variable overrides resolve in the approved order.

Execution: run the unit suite that loads controlled fixtures for each precedence layer and asserts the effective values.

Automation: Required.

Command pattern: dotnet test tests/<core-test-project> --filter UTP1_01

Expected evidence: test output shows the effective-value assertions for default-only, template-plus-local-settings, secret-file, and environment-override cases all passing.

UTP1-02: Settings File Read and Write

Verify that non-secret operator settings persist to settings.json and reload without schema drift or silent loss of values.

Execution: write representative settings to a temporary file, reload them through the Phase 1 settings component, and compare the effective model.

Automation: Required.

Command pattern: dotnet test tests/<core-test-project> --filter UTP1_02

Expected evidence: persisted values and reloaded values match, and no checked-in template file is modified.

UTP1-03: Secret Presence Detection

Verify that missing, partial, and present secret states are detected correctly without exposing secret contents.

Execution: exercise the detection logic with empty, partial, complete, and environment-override fixtures.

Automation: Required.

Command pattern: dotnet test tests/<core-test-project> --filter UTP1_03

Expected evidence: the component reports the correct state class and never includes raw secret values in assertion output.

UTP1-04: Redaction and Error Safety

Verify that logs, validation messages, and exception paths redact or omit secret values and sensitive file contents.

Execution: inject known sentinel values into secret fixtures, force validation and parse errors, and assert that captured logs/messages do not contain the sentinel values.

Automation: Required.

Command pattern: dotnet test tests/<core-test-project> --filter UTP1_04

Expected evidence: sentinel values do not appear in captured logs, messages, or serialized error objects.

UTP1-05: Shell Navigation State

Verify that the shell can select and retain each approved top-level view through centralized navigation state.

Execution: instantiate the shell-navigation model and iterate through all approved destinations.

Automation: Required.

Command pattern: dotnet test tests/<app-test-project> --filter UTP1_05

Expected evidence: all eleven approved destinations are recognized and retained without duplicate or orphaned entries.

UTP1-06: Placeholder State Semantics

Verify that placeholder, inactive, not configured, and not connected states are distinguishable in view-model state and do not collapse into the same generic condition.

Execution: construct view-model states for each approved condition and assert that their labels, flags, and severity classifications differ as intended.

Automation: Required.

Command pattern: dotnet test tests/<app-test-project> --filter UTP1_06

Expected evidence: test output shows distinct state mappings for each condition and no ambiguous fallback state.

UTP1-07: Policy Edit Persistence

Verify that approved candidate notes, watchlist notes, protected-asset modes, and other non-secret policy edits persist as local operator state.

Execution: mutate representative policy objects, save them through the local settings path, reload them, and compare the effective policy state.

Automation: Required.

Command pattern: dotnet test tests/<core-test-project> --filter UTP1_07

Expected evidence: approved candidate notes, DOGE watchlist note, and protected-asset mode values round-trip without loss.

UTP1-08: Risk Profile Transition Rules

Verify that preset profiles remain read-only defaults, Custom is the only direct-edit mode, unsaved overwrite confirmation is required, and Reset to Default restores the recommended state.

Execution: exercise profile transitions, edit attempts under preset mode, overwrite confirmation logic, and reset flows in the threshold-state model.

Automation: Required.

Command pattern: dotnet test tests/<app-test-project> --filter UTP1_08

Expected evidence: preset-edit attempts are rejected, overwrite confirmation is requested when required, and reset restores the approved recommended baseline.

UTP1-09: Source Registry State Model

Verify that the reserved source states Active, Observed, Ignored, Blocked, and Needs Review are represented consistently and reversible actions map to the correct target state.

Execution: run state-transition tests for each reserved source state and each reversible action.

Automation: Required.

Command pattern: dotnet test tests/<app-test-project> --filter UTP1_09

Expected evidence: all reserved states exist in the model, invalid transitions fail, and reversible actions land in the correct target state.

UTP1-10: AI-Created Watcher Defaults

Verify that AI-created watchers enter the registry in Observed or Needs Review rather than an implicitly trusted active state.

Execution: construct watcher-creation inputs flagged as AI-created and assert their default state and trust flags.

Automation: Required.

Command pattern: dotnet test tests/<app-test-project> --filter UTP1_10

Expected evidence: no AI-created watcher defaults to Active.

UTP1-11: Quote and Valuation Defaults

Verify that USD remains the locked Phase 1 valuation currency and preferred initial routing posture while USDC remains recognized as an approved secondary quote/routing asset.

Execution: assert default configuration/model values for valuation currency, preferred route, and approved secondary route.

Automation: Required.

Command pattern: dotnet test tests/<core-test-project> --filter UTP1_11

Expected evidence: valuation currency resolves to USD, USD is preferred, and USDC is allowed but not primary.

UTP1-12: Manual Update Result Mapping

Verify that manual update-check responses from GitHub Releases map cleanly to current-version, latest-version, update-available, and no-update states without implying self-update capability.

Execution: replay release-source fixtures for no release, current version, newer version, malformed response, and network-failure scenarios.

Automation: Required.

Command pattern: dotnet test tests/<app-test-project> --filter UTP1_12

Expected evidence: valid responses map correctly, malformed responses surface understandable failures, and no state implies automatic installation.

UTP1-13: Invalid Settings and Secret Fixtures

Verify that malformed settings.json, malformed secrets.env, and missing required keys produce understandable validation outcomes and do not crash the process.

Execution: load intentionally invalid fixture files through the validation components and assert blocking-failure, warning, or informational classification as appropriate.

Automation: Required.

Command pattern: dotnet test tests/<core-test-project> --filter UTP1_13

Expected evidence: each invalid fixture produces the expected validation classification and redacted message text.

UTP1-14: Unsupported Surface Metadata Exclusion

Verify that shell navigation metadata and top-level view registration do not include bank funding, withdrawal, wallet transfer, staking, Web3, margin, borrowing, or derivatives surfaces.

Execution: inspect the registered Phase 1 destinations and supported command identifiers in component-level tests.

Automation: Required.

Command pattern: dotnet test tests/<app-test-project> --filter UTP1_14

Expected evidence: unsupported account-action surfaces are absent from navigation and command registries.

Execution Method

  • Implement automated unit tests in the Phase 1 test project structure under tests/.
  • Prefer pure component and view-model tests over UI automation for this layer.
  • Use controlled test doubles for file-system, release-source, and secret-source interactions where practical.
  • Maintain stable test naming so the documented --filter commands remain usable as focused execution commands.
  • Record unit-test results in a durable artifact under docs/test-results/ when Phase 1 execution begins.

Exit Criteria

  • All critical Phase 1 state-management and configuration contracts have dedicated unit coverage.
  • Secrets and redaction behavior are validated at the component level.
  • No placeholder-state, negative validation path, or profile-transition rule remains untested.
  • Unsupported account-action surfaces are protected by automated component-level tests.
