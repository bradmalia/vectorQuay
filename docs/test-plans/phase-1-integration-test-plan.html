Integration Test Plan

Phase 1 Integration Test Plan

Cross-layer validation for the repository baseline, Avalonia shell, local settings model, and approved Phase 1 operator flows.

Versionv1.0
StatusApproved
OwnerBrad Malia
ThemeIndigo Signal

Purpose

This plan verifies that the Phase 1 implementation works as a coherent baseline across repository structure, configuration loading, shell startup, view navigation, persistence, and release-check behavior.

Test Goal

Prove that the approved Phase 1 baseline is runnable, safe, and structurally ready for Phase 2 Coinbase read-only integration.

Scope

  • Validate repository and project-structure completeness.
  • Validate shell startup on Ubuntu-first local workflow.
  • Validate local settings and external-secret boundaries.
  • Validate navigation across all approved top-level screens.
  • Validate policy/configuration persistence and placeholder-state behavior.
  • Validate real manual GitHub Releases update checks.

Out of Scope

  • Live Coinbase API integration.
  • Live order placement or cancellation.
  • Paper trading.
  • Outbound email or SMS delivery.
  • Automatic asset-discovery jobs.
  • Self-updating installation behavior.

Integration Test Areas

  • Repository-to-build workflow alignment.
  • Configuration template to live settings-path alignment.
  • Secret-source detection to UI status alignment.
  • Shell navigation to approved PRD mock structure alignment.
  • Policy and settings edits to persistence-layer alignment.
  • Version display to release-check alignment.
  • Approved product boundaries to shell affordance alignment.
  • Ignored-file enforcement and repo-safety alignment.
  • Validation-command behavior and evidence capture alignment.

Execution Environment and Fixtures

  • Primary execution environment: Ubuntu-first local workstation with .NET 8 SDK installed and the Phase 1 solution present.
  • Primary command baseline: dotnet restore, dotnet build, dotnet test, and the documented shell run command such as dotnet run --project src/VectorQuay.App.
  • Required fixture set: clean repo checkout, temporary local settings directory, temporary secrets.env variations, environment-variable override fixture, placeholder release-source fixture or controlled GitHub Releases test target, and representative non-secret policy/configuration edits.
  • Until the exact integration-test project name exists, any dotnet test command that references an integration-test project should be interpreted as a command pattern using the eventual integration-test project path under tests/.
  • Windows-compatibility strategy without a Windows machine: treat this as a structural compatibility check rather than a full runtime compatibility test, require a RID-neutral project design review, require successful non-Windows build/test execution, and when the implementation exists, run a compatibility smoke command such as dotnet publish src/VectorQuay.App -c Release -r win-x64 --self-contained false if the chosen packaging path supports it.
  • Expected evidence: saved command transcript, build/test summaries, screenshots or textual captures of key shell states, and a durable result record under docs/test-results/phase-1-integration-results.md or equivalent.

Test Cases

ITP1-01: Repository Scaffold Integrity

Verify that the repository contains the approved source, test, docs, assets, and configuration-template structure.

Automation: Required.

Pass condition: project layout matches the approved Phase 1 baseline without collapsing core and app concerns into a single unmanaged structure.

Expected evidence: captured repository checklist mapped to the approved structure.

ITP1-02: Ubuntu Restore, Build, and Run

Verify that the documented Ubuntu workflow restores dependencies, builds successfully, and launches the shell application.

Automation: Required for restore and build; shell launch may be verified through automation or a controlled manual run until UI automation exists.

Command: dotnet restore, dotnet build, and the documented shell launch command.

Pass condition: no ad hoc undocumented steps are required.

Expected evidence: captured command output showing successful restore, build, and launch.

ITP1-03: Safe Startup With Missing Secrets

Verify that the application starts with no secrets present and presents an understandable not-configured state rather than failing hard.

Automation: Required for startup/validation classification; UI-state capture may be manual until shell automation exists.

Fixture: no secrets.env file and no relevant environment variables.

Command: documented shell launch command plus any documented validation command.

Pass condition: shell launch succeeds and secret-source status is visible without revealing secret values.

Expected evidence: startup output and UI capture or textual state record showing not-configured status.

ITP1-04: Secret File and Environment Override Integration

Verify that external secret-file values are recognized and same-name environment variables override them in the effective runtime state.

Automation: Required.

Fixture: temporary secrets.env plus process-scoped environment variables with conflicting values.

Command pattern: documented validation command and focused integration test command such as dotnet test tests/<integration-test-project> --filter ITP1_04.

Pass condition: effective configuration reflects the approved precedence contract.

Expected evidence: result artifact records the resolved source precedence without exposing raw secrets.

ITP1-05: Top-Level Navigation Coverage

Verify that Home, Configuration, Assets, Policies, Portfolio, Activity, Performance, Risk & Thresholds, Sources, Alerts, and About all render through the shared shell.

Automation: Required for destination enumeration; guided manual verification is acceptable for visual rendering until UI automation exists.

Command: shell launch command plus focused navigation automation or guided manual run with a checklist artifact.

Pass condition: each screen is reachable and stable through centralized navigation.

Expected evidence: navigation checklist with pass/fail entry for every approved top-level destination.

ITP1-06: Policy and Configuration Persistence

Verify that non-secret configuration and policy edits persist across application restart through the application-managed local settings file.

Automation: Required for persistence verification; UI interaction may be manual until automation exists.

Fixture: representative configuration changes plus policy changes for approved candidate notes, DOGE watchlist note, and protected-asset modes.

Command: shell run, save action, restart, and focused persistence verification command or checklist.

Pass condition: edits remain after restart and checked-in templates remain unchanged.

Expected evidence: before/after state capture and confirmation that only the temporary application-managed settings file changed.

ITP1-07: Reserved-State Presentation

Verify that inactive, placeholder, not configured, and not connected states remain visually and semantically distinct across the shell.

Automation: Required for state classification; manual capture is acceptable for the rendered visual proof until UI automation exists.

Fixture: controlled startup modes that trigger each approved state.

Pass condition: the shell does not imply live trading, paper trading, or live source ingestion is active.

Expected evidence: state checklist or screenshots/text captures showing each condition distinctly.

ITP1-08: Unsupported Account-Surface Exclusion

Verify that the shell does not expose supported flows for bank funding, withdrawals, payment-method changes, external wallet transfers, staking/Web3, margin, borrowing, or derivatives.

Automation: Required for navigation/registration checks; manual screen inspection is acceptable as supplemental evidence.

Pass condition: no Phase 1 screen contradicts the approved trade-only shell boundary.

Expected evidence: reviewed navigation map and screen checklist showing these surfaces are absent.

ITP1-09: Source Registry Contract

Verify that the Sources view preserves the locked tabs, state model, selection behavior, and watcher/direct-source registry treatment.

Automation: Required for state and selection assertions; manual review may supplement rendering evidence.

Pass condition: sources and watchers share one registry model and AI-created watchers default to a non-trusted state.

Expected evidence: source-registry walkthrough record showing tabs, selection behavior, and default watcher state.

ITP1-10: Manual GitHub Releases Update Check

Verify that the About view performs a real manual check against the configured GitHub Releases source and presents current version, latest available version, and download/open affordances.

Automation: Required for release-response handling; manual About-screen verification is acceptable for rendered state until UI automation exists.

Fixture: configured GitHub Releases test target or controlled response fixture.

Command: shell launch plus manual update-check action; supplement with focused integration command such as dotnet test tests/<integration-test-project> --filter ITP1_10 where available.

Pass condition: update discovery works without silent install or forced restart behavior.

Expected evidence: About view capture or textual state record with current version, latest version, and manual download/open affordance.

ITP1-11: Branding and Version Consistency

Verify that branding, owner attribution, and coherent version display appear consistently in shell metadata, Home, and About surfaces.

Automation: Required for version-value consistency where accessible programmatically; manual evidence is acceptable for rendered branding review.

Pass condition: no screen or build artifact presents conflicting version or attribution information.

Expected evidence: artifact record showing matching title/window metadata, Home version, and About version values.

ITP1-12: Ignored-File Enforcement

Verify that the application-managed settings file and external secret file are covered by the repo’s ignore rules and do not appear as normal commit candidates.

Automation: Required.

Fixture: temporary creation of representative local settings and secret files in the approved paths or mapped test-equivalent paths.

Command: git status --short after fixture creation, plus any documented ignore-verification step.

Pass condition: ignored local files do not appear as tracked or addable artifacts in normal workflow.

Expected evidence: captured git status --short output showing no accidental inclusion of the protected files.

ITP1-13: Save, Validate, and Reset Command Flows

Verify that meaningful operator actions required by the Phase 1 shell contract perform the correct behavior and surface the correct result states.

Automation: Required for underlying behavior verification; manual UI interaction is acceptable for rendered confirmation until UI automation exists.

Fixture: editable non-secret configuration, editable policy state, and threshold-profile state with unsaved changes.

Command: shell launch with focused interaction checklist or automation covering Save, Validate, Reset to Default, and overwrite-confirmation flows.

Pass condition: Save persists correctly, Validate returns understandable classifications, Reset restores defaults, and overwrite confirmation appears when required.

Expected evidence: interaction log or captured checklist with outcome for each operator action.

ITP1-14: Negative Validation Paths

Verify that malformed settings.json, malformed secrets.env, and invalid release-source responses surface understandable failures rather than crashes.

Automation: Required.

Fixture: intentionally invalid settings, invalid secret file, and invalid GitHub Releases payload.

Command pattern: documented validation command and focused integration test command such as dotnet test tests/<integration-test-project> --filter ITP1_14.

Pass condition: failures are classified and understandable, and the shell or validator remains controlled.

Expected evidence: result artifact captures each negative path with classification and redacted message text.

ITP1-15: Windows Compatibility Smoke Strategy

Verify Windows compatibility to the extent possible without a Windows machine by treating this as a structural compatibility check and, when supported by the implementation, performing a Windows RID publish smoke test.

Automation: Required for the publish smoke command when applicable; structural review is required in all cases.

Command: dotnet publish src/VectorQuay.App -c Release -r win-x64 --self-contained false after the implementation exists, or document why the command is not yet applicable.

Pass condition: the project structure and dependency choices do not show an obvious Windows-blocking assumption, and the publish smoke test succeeds if the project has reached a stage where publish is applicable.

Expected evidence: publish output or an explicit documented exception describing why only structural compatibility evidence is available at the current implementation stage.

Execution Method

  • Perform repository and runtime validation against the approved Phase 1 PRD and TRD.
  • Use a controlled local test environment with temporary settings and secret sources outside the repo.
  • Capture the exact commands used in the result artifact so the validation is repeatable by a later session.
  • Capture pass/fail outcomes and defects in a durable result artifact under docs/test-results/.
  • Resolve any integration contradiction before Phase 1 is considered complete.

Exit Criteria

  • The documented build and run workflow succeeds on the primary environment.
  • All approved Phase 1 views render through the shell and preserve the required states and boundaries.
  • Configuration, persistence, validation, ignored-file enforcement, and manual update-check flows work coherently end to end.
  • The executed commands, fixtures, and evidence are recorded clearly enough for a later session to reproduce the run.
