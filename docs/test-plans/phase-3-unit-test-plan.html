Unit Test Plan

Phase 3 Unit Test Plan

Persistence-model, recovery, and audit normalization coverage for the storage phase.

Execution Environment

  • Primary environment: local .NET 8 solution on the Ubuntu-first development workstation.
  • Primary command baseline: dotnet restore, dotnet build, and dotnet test.
  • Required fixtures: valid normalized snapshot fixture, corrupted persisted-state fixture, missing-file fixture, audit-event creation fixture, and retention-boundary fixture.
  • Expected evidence: automated test output, targeted failure messages, and durable result notes under docs/test-results/.

Required Unit Coverage

  • UTP3-01: persisted snapshot serialization/deserialization round-trip.
  • UTP3-02: schema-version or compatibility fallback behavior.
  • UTP3-03: corruption-safe fallback when stored files are malformed.
  • UTP3-04: audit/event record creation for policy, alert, and source changes.
  • UTP3-05: startup recovery chooses restored state without implying live refresh success.
  • UTP3-06: secret values are excluded from persisted event/snapshot payloads.
  • UTP3-07: retention and truncation logic where bounded history is required.

Execution Contract

UTP3-01: Snapshot Round-Trip

Automation: Required.

Fixture: normalized persisted snapshot with balances, holdings summary, and refresh metadata.

Pass condition: serialized output can be read back without losing required fields or changing semantic values.

UTP3-03: Corruption Fallback

Automation: Required.

Fixture: malformed JSON or incompatible stored payload.

Pass condition: the storage layer returns a safe fallback classification and does not throw an unhandled exception into the shell path.

UTP3-05: Restored-State Truthfulness

Automation: Required.

Fixture: recovered snapshot plus failed/absent live refresh result.

Pass condition: restored-state classification remains distinct from live-state classification.

UTP3-06: Secret Exclusion

Automation: Required.

Fixture: representative settings, snapshot, and audit-event payloads with secret-adjacent fields available in memory.

Pass condition: persisted payloads contain no secret values, secret file contents, or raw credential strings.

Pass Condition

Persisted models are stable, recovery is safe under normal and corrupted-state conditions, and audit/event creation is deterministic enough to support later debugging and evaluation.
