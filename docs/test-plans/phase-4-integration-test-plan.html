Integration Test Plan

Phase 4 Integration Test Plan

Cross-layer validation for deterministic decisions, GUI visibility, and non-bypassable risk approval.

Execution Environment and Fixtures

  • Primary execution environment: Ubuntu-first local workstation with valid Phase 3 durable-state behavior and valid Coinbase read-only connectivity.
  • Primary command baseline: dotnet restore, dotnet build, dotnet test, and dotnet run --project src/VectorQuay.App.
  • Required fixtures: valid live Coinbase snapshot, persisted durable state, approved asset universe, protected-asset example, ineligible-asset example, and multiple risk-profile examples.
  • Expected evidence: command summaries, strategy/risk walkthrough captures, and result artifacts under docs/test-results/.

Required Integration Gates

  • ITP4-01: universe eligibility runs at the approved times and gates which assets enter candidate evaluation.
  • ITP4-02: deterministic candidate decisions appear from current live read-only data.
  • ITP4-03: protected assets and hard limits reject candidates visibly and durably.
  • ITP4-04: aggressiveness profile changes alter decision posture but do not bypass hard risk.
  • ITP4-05: Activity distinguishes deterministic decision records from true account-event history.
  • ITP4-06: decision outputs are durable across restart through the Phase 3 persistence layer.
  • ITP4-07: no execution or paper-trading path is introduced.

Execution Contract

ITP4-01: Eligibility Timing and Gating

Automation: Required where logic is isolated; manual shell review may supplement UI evidence.

Fixture: mix of eligible, ineligible, and protected assets plus current approved-universe state.

Command: launch the shell, perform startup refresh, trigger Refresh Universe, then change policy state affecting one asset and observe reevaluation.

Pass condition: only eligible assets enter candidate evaluation, and reevaluation occurs at the documented times with durable explanation/audit visibility.

Expected evidence: startup state capture, post-refresh universe change capture, and an audit/history record showing why an asset was eligible, blocked, or unchanged.

ITP4-02: Deterministic Candidate Emergence

Automation: Required where logic is isolated; manual shell review may supplement UI evidence.

Fixture: valid live Coinbase snapshot with at least one asset that should score into the top candidate set.

Command: launch the shell, allow startup/live refresh to complete, then inspect Overview and Activity.

Pass condition: the same inputs produce the same candidate posture, and the candidate appears in operator-facing views with deterministic explanation context.

Expected evidence: repeated run capture or logged decision payload showing stable decision output from the same fixture.

ITP4-03: Hard-Risk Rejection

Automation: Required where logic is isolated; manual shell review may supplement UI evidence.

Fixture: asset that scores well but violates protected mode or concentration/position rules.

Command: evaluate the candidate through the live shell after refresh and inspect Activity / Risk & Thresholds output.

Pass condition: the candidate is visibly rejected or constrained with explicit hard-risk reasons and durable decision history.

Expected evidence: Activity decision record plus Risk & Thresholds state showing the blocking rule and final constrained/rejected disposition.

ITP4-04: Profile Change Without Hard-Risk Bypass

Automation: Required.

Fixture: same asset/account state under Conservative, Balanced, and Aggressive profiles.

Command: switch profiles and reevaluate decisions.

Pass condition: score posture or ranking sensitivity may change, but hard-limit failures remain blocking.

Expected evidence: three decision outputs under the three profiles, showing changed soft posture without hard-limit drift.

ITP4-05: Decision Ledger Separation

Automation: Required where practical; manual shell review may supplement UI evidence.

Fixture: retained account-event history plus new deterministic decisions.

Command: review Activity after refresh and after restart.

Pass condition: deterministic decisions remain clearly distinguishable from real executed account events.

Expected evidence: Activity capture showing separate decision records vs account-event records without ambiguous labeling.

ITP4-06: UI Explainability Completeness

Automation: Manual shell review is acceptable for this gate in Phase 4.

Fixture: one approved candidate, one constrained candidate, and one rejected candidate.

Command: inspect the Activity decision-explanation surface for each case.

Pass condition: the operator can see eligibility result, score breakdown, hard-risk checks, final disposition, and size rationale where applicable.

Expected evidence: decision-inspection capture for all three cases showing complete explanation sections.

ITP4-07: No Execution / No Paper Drift

Automation: Required.

Fixture: fully functioning deterministic decision engine with current Phase 4 UI surfaces enabled.

Command: review available commands and runtime behavior after startup and manual reevaluation.

Pass condition: no execution or paper-trading path is introduced.

Expected evidence: command/UI review record showing deterministic decisioning only, with no order lifecycle or simulated fill path available.

ITP4-08: Evaluation Progress Visibility

Automation: Manual shell review is acceptable for this gate in Phase 4.

Fixture: startup refresh plus an operator-triggered Refresh Universe reevaluation.

Command: observe footer/status bar and action surfaces while evaluation runs.

Pass condition: the operator can see that evaluation is in progress, distinguish the current stage, and see a completion timestamp or equivalent completion cue.

Expected evidence: runtime capture or manual result note showing visible transition through evaluation states.

Pass Condition

The product can generate deterministic candidate decisions, show them clearly in the shell, and enforce hard-risk outcomes without weakening the read-only boundary or blurring decisions with actual executed account history.
