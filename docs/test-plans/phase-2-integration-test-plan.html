Integration Test Plan

Phase 2 Integration Test Plan

Cross-layer validation for shell rendering, Coinbase read-only data flow, and capital restriction confidence.

Versionv0.1
StatusDraft baseline
OwnerBrad Malia
ThemeIndigo Signal

Purpose

Phase 2 integration testing should prove that Coinbase read-only data flows coherently through the existing shell and that the product understands account truth, available capital, and restrictions before any execution work begins.

Execution Environment and Fixtures

  • Primary execution environment: Ubuntu-first local workstation with the Phase 2 .NET 8 solution and valid or intentionally invalid Coinbase credential fixtures available.
  • Primary command baseline: dotnet restore, dotnet build, dotnet test, and the documented shell launch command such as dotnet run --project src/VectorQuay.App.
  • Required fixture set: valid Coinbase read-only credentials, missing-credential fixture, incomplete-credential fixture, rejected-auth fixture where possible, representative account/product response fixtures, and a known local policy overlay state.
  • Expected evidence: command summaries, walkthrough captures for affected screens, refresh-state records, and a durable result artifact under docs/test-results/phase-2-integration-results.md or equivalent.

Required Integration Gates

  • ITP2-01: Successful read-only Coinbase authentication and connection-health display.
  • ITP2-02: Accurate account-balance rendering in Overview and Portfolio.
  • ITP2-03: Accurate product-universe rendering in Assets with approved/watch overlays.
  • ITP2-04: Existing-funds-only capital interpretation with no unsupported funding paths implied.
  • ITP2-05: Policy restrictions remain visible and coherent alongside real Coinbase data.
  • ITP2-06: Manual refresh updates the shell cleanly without crashes or inconsistent state.
  • ITP2-07: Missing credentials and auth failures surface understandable operator errors.
  • ITP2-08: The shell still presents an inactive trading state despite live read-only data.
  • ITP2-09: Startup refresh occurs once when credentials are complete and does not become a silent polling loop.

Execution Contract

ITP2-01: Read-Only Coinbase Authentication

Verify that the shell connects successfully to Coinbase using the approved read-only credential path and surfaces connection health clearly.

Automation: Required for underlying service verification; manual UI review is acceptable for rendered proof until UI automation exists.

Fixture: valid read-only Coinbase credentials configured through the approved secret model.

Command: documented shell launch command plus focused integration command pattern such as dotnet test tests/<integration-test-project> --filter ITP2_01.

Pass condition: the app reaches a truthful connected state without exposing secret values or implying write capability.

Expected evidence: connected-state capture showing connection health, last refresh state, and redacted credential-source status.

ITP2-02: Accurate Balance Rendering

Verify that Overview and Portfolio show truthful current balances, holdings, and value interpretations from Coinbase read-only data.

Automation: Required for mapping verification; manual shell review may supplement rendering evidence.

Fixture: valid read-only Coinbase account with stable known balances or controlled integration fixture.

Expected evidence: captured shell state or textual validation record showing account totals, holdings, and current-value consistency.

Pass condition: shell values match the verified read-only account or controlled fixture output without contradictory totals.

ITP2-03: Asset Registry Overlay Integrity

Verify that Assets merges Coinbase product visibility with local approved-candidate, watchlist, and restricted overlays without losing local policy meaning.

Automation: Required.

Fixture: live or controlled product set containing approved assets, DOGE, and non-approved products with a known local policy overlay state.

Command: shell launch plus refresh action and focused integration command pattern such as dotnet test tests/<integration-test-project> --filter ITP2_03.

Pass condition: Coinbase refresh does not erase or silently reinterpret local overlay state.

Expected evidence: before/after overlay record showing policy state intact after refresh.

ITP2-04: Existing-Funds-Only Capital Interpretation

Verify that the shell derives available capital only from existing Coinbase balances and does not imply unsupported funding paths.

Automation: Required.

Fixture: known balance state with distinguishable available cash and held assets.

Expected evidence: walkthrough or captured state showing balance-derived capital interpretation and absence of off-platform funding surfaces.

Pass condition: no visible shell state suggests deposits, external funding, or unsupported transfer sources are part of available capital.

ITP2-05: Policy Overlay Preservation

Verify that local policy restrictions remain visible and authoritative when live Coinbase data is loaded.

Automation: Required.

Fixture: known local policy state including approved candidates, DOGE watchlist note, and protected-asset modes.

Command: refresh live Coinbase data, revisit Assets and Policies, and run any focused integration verification command.

Pass condition: refresh does not overwrite approved candidates, DOGE watchlist posture, or protected-asset modes.

Expected evidence: before/after state capture showing local policy values unchanged.

ITP2-06: Manual Refresh Behavior

Verify that manual refresh updates shell state coherently, including last-refresh cues and connection-health updates, without crashes or stale-state confusion.

Automation: Required for service behavior; manual shell interaction may supplement evidence.

Fixture: valid credentials and a shell state loaded before manual refresh is triggered.

Command: shell launch, trigger manual refresh, and capture the resulting state transition.

Pass condition: refresh completes or fails safely with distinct state transitions and no UI contradiction.

Expected evidence: before/after refresh record showing timestamp or connection-state change.

ITP2-07: Credential and Auth Failure Handling

Verify that missing credentials, incomplete credentials, and auth failures produce understandable operator errors and safe non-connected shell state.

Automation: Required.

Fixture: missing-credential, incomplete-credential, and rejected-auth variants.

Pass condition: each failure mode is distinguishable enough for operator troubleshooting and none crashes the shell.

Expected evidence: error-state captures or textual result log for each failure mode.

ITP2-08: Inactive-Trading Boundary Preservation

Verify that the shell continues to present an inactive trading state even while live read-only Coinbase data is visible.

Automation: Required for baseline assertions; manual visual confirmation may supplement evidence.

Command: shell launch with successful read-only Coinbase refresh and focused navigation review of Overview, Portfolio, Assets, and Activity.

Pass condition: no screen implies order placement, active strategy, or execution readiness.

Expected evidence: checklist showing inactive-trading language and absence of execution affordances across relevant screens.

ITP2-09: One-Time Startup Refresh Without Polling

Verify that when a complete Coinbase credential set is present, the shell performs one safe startup refresh and does not enter an unapproved silent retry or polling loop.

Automation: Required.

Fixture: complete valid Coinbase credentials and observable refresh-state logging or shell state transitions.

Command: shell launch with startup refresh instrumentation and focused integration verification command such as dotnet test tests/<integration-test-project> --filter ITP2_09.

Pass condition: one startup refresh attempt occurs, the resulting state is visible, and no silent recurring refresh loop begins.

Expected evidence: startup-state record showing one refresh attempt and no background polling behavior.

Required Evidence

  • Manual or automated walkthrough record for Overview, Configuration, Assets, Portfolio, Sources, and About.
  • Before/after refresh state capture showing last-refresh and connection-health updates.
  • Text or screenshot evidence that unsupported funding and non-trade account surfaces are absent.
  • Evidence that local policy overlays remain intact after live refresh.

Pass Condition

Read-only Coinbase data is accurate enough for operator trust, UI state remains coherent under success and failure conditions, and no integration path creates accidental execution capability.
