Phase 1 Regression Test Plan

Baseline-preservation plan for approved Phase 1 shell behavior, configuration rules, and Phase 0 carry-forward constraints.

Versionv1.0
StatusApproved
OwnerBrad Malia
ThemeIndigo Signal

Purpose

This plan protects the approved Phase 1 baseline from accidental scope expansion, policy drift, or weakening of Phase 0 carry-forward constraints during implementation and later revisions.

Test Goal

Prevent the Phase 1 shell and configuration baseline from silently changing into something broader, riskier, or less reusable than the approved documents allow.

Regression Protection Areas

  • Coinbase Advanced only baseline.
  • Spot-only and trade-only shell posture.
  • No bank, wallet, staking, Web3, margin, or derivatives surfaces.
  • Zero-secrets-in-repo and BYO-credentials support.
  • USD valuation and preferred initial routing posture.
  • USDC secondary routing/defensive allowance.
  • Locked local settings paths and precedence order.
  • Approved candidate class and DOGE watchlist treatment.
  • Dynamic discovery later without all-assets-by-default live eligibility.
  • Protected-asset and local-operator-policy boundaries.
  • Manual-only update flow and no self-updating behavior.

Execution Environment and Evidence

  • Run regression review whenever Phase 1 requirements, shell structure, config paths, or protected policy areas change materially.
  • Primary review inputs: approved roadmap, Phase 0 decision memo, approved Phase 1 PRD, approved Phase 1 TRD, and the latest implementation/test result artifacts.
  • Expected evidence: comparison checklist, identified drift items if any, approval notes for intentional changes, and a durable result record under docs/test-results/phase-1-regression-results.md or equivalent.

Regression Test Cases

RTP1-01: Exchange-Surface Preservation

Verify that the Phase 1 baseline does not introduce a non-Coinbase exchange path or imply multi-exchange support without explicit approval.

RTP1-02: Trade-Only Shell Preservation

Verify that later revisions do not introduce operator flows for account actions outside the approved trade-only boundary.

RTP1-03: Secrets and Template Preservation

Verify that real credentials never move into the repo, checked-in templates do not become live settings sources, and redaction protections remain intact.

RTP1-04: Quote and Valuation Rule Preservation

Verify that USD remains the locked Phase 1 valuation currency and preferred initial routing posture unless a later approved scope change explicitly alters it.

RTP1-05: USDC Secondary Allowance Preservation

Verify that later revisions do not silently remove USDC as an approved secondary quote/routing asset without explicit approval.

RTP1-05A: Settings Path and Precedence Preservation

Verify that the locked settings.json path, secrets.env path, and the approved precedence order including environment-variable overrides are not silently changed.

RTP1-06: Asset-Universe Baseline Preservation

Verify that the approved initial candidate class and DOGE watchlist note remain visible and are not replaced by an unrestricted all-assets posture.

RTP1-07: Dynamic Discovery Boundary Preservation

Verify that future dynamic discovery remains reserved without implying automatic live tradability for newly discovered assets.

RTP1-08: Protected-Asset Boundary Preservation

Verify that local policy editing does not weaken the requirement that later technical enforcement layers remain authoritative.

RTP1-09: Source Registry State Preservation

Verify that the locked source-state model and reversible state actions are not weakened or collapsed in later revisions.

RTP1-10: Update Flow Boundary Preservation

Verify that the manual GitHub Releases update check does not expand into silent self-update, background install, or forced restart behavior without explicit approval.

RTP1-11: Placeholder-State Preservation

Verify that Phase 1 placeholders continue to communicate inactive or future-ready status rather than implying real trading or live-ingestion capability.

RTP1-12: Resale-Friendly Baseline Preservation

Verify that later revisions do not introduce one-operator, one-machine assumptions that undermine future resale or reusable installation goals.

Execution Method

  • Run regression review whenever the Phase 1 baseline or any protected policy area changes materially.
  • Compare new revisions against the approved roadmap, Phase 0 carry-forward decisions, Phase 1 PRD, and Phase 1 TRD baselines.
  • Where implementation exists, compare current behavior and test-result evidence against the last approved result set rather than relying on document review alone.
  • Store results in a durable review artifact under docs/test-results/.
  • Reject any silent scope expansion or weakened control boundary that lacks explicit approval.

Exit Criteria

  • All protected Phase 1 baseline decisions remain intact.
  • No later revision weakens Phase 0 carry-forward constraints without explicit approval.
  • Any approved change to the baseline remains documented and traceable.
