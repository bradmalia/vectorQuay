Integration Test Plan

Phase 3 Integration Test Plan

Cross-layer validation for durable state, recovery, and replayable history.

Execution Environment and Fixtures

  • Primary execution environment: Ubuntu-first local workstation with the Phase 3 .NET 8 solution and valid Phase 2 Coinbase read-only connectivity.
  • Primary command baseline: dotnet restore, dotnet build, dotnet test, and the documented shell launch command such as dotnet run --project src/VectorQuay.App.
  • Required fixtures: valid last-known snapshot fixture, missing-storage fixture, corrupted-storage fixture, valid Coinbase read-only credentials, and known local policy/source/alert-change actions.
  • Expected evidence: command summaries, restart walkthrough captures, recovery-state records, and durable results under docs/test-results/phase-3-integration-results.md or equivalent.

Required Integration Gates

  • ITP3-01: startup recovers last-known state safely before the next refresh completes.
  • ITP3-02: a successful Coinbase refresh updates both live shell state and durable snapshot state.
  • ITP3-03: policy edits, alert-rule edits, and source changes create durable audit records.
  • ITP3-04: corrupted or missing persisted state falls back safely without crashing the shell.
  • ITP3-05: Activity consumes durable history across restarts.
  • ITP3-06: the read-only/no-polling boundary remains intact after persistence is introduced.

Execution Contract

ITP3-01: Startup Recovery

Automation: Required where practical; manual shell review may supplement startup-state rendering evidence.

Fixture: valid persisted snapshot from a prior successful live refresh.

Command: launch the shell with recovery data present and capture pre-refresh and post-refresh states.

Pass condition: last-known state is restored safely, labeled as restored, and replaced or preserved correctly after the next live refresh outcome.

ITP3-02: Refresh Updates Durable State

Automation: Required.

Fixture: valid read-only Coinbase credentials plus an existing older persisted snapshot.

Command: launch the shell, trigger a successful live refresh, then inspect the durable snapshot/state files and the rendered shell state.

Pass condition: the rendered live state and the updated durable snapshot reflect the same new refresh timestamp and normalized summary values.

Expected evidence: before/after durable snapshot record plus shell capture showing consistent updated state.

ITP3-03: Durable Audit Creation

Automation: Required.

Fixture: known operator actions such as policy apply, alert-rule save, and source add/remove.

Pass condition: each action produces a durable audit/event record with required identifying fields.

ITP3-04: Corruption Fallback

Automation: Required.

Fixture: malformed persisted snapshot/history files.

Pass condition: the shell starts safely, surfaces the failure understandably, and avoids crashing or presenting malformed data as valid.

ITP3-05: Activity Uses Durable History Across Restarts

Automation: Required where practical; manual shell review may supplement UI evidence.

Fixture: stored recent activity history from a prior session plus no new live events on the next startup.

Command: populate durable activity history, restart the app, and inspect the Activity screen before and after a live refresh attempt.

Pass condition: Activity shows the persisted recent history across restart and labels the truth state correctly until new live data changes it.

Expected evidence: restart walkthrough showing durable activity continuity with correct restored/live labeling.

ITP3-06: Read-Only / No-Polling Preservation

Automation: Required.

Fixture: valid Phase 3 persistence plus valid read-only Coinbase credentials.

Pass condition: persistence introduces no write-capable path and no recurring silent refresh loop.

Pass Condition

The product can restart safely, reconstruct recent operational state, and expose audit history without weakening the Phase 2 exchange boundary.
