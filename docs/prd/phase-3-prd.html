Executive Summary

Phase 3 turns the Phase 2 read-only shell into a durable operational baseline. It adds local persistence for last-known account state, bounded portfolio-value history, operator policy state, and a replayable audit trail so the app can survive restart, explain prior state, and support later strategy and paper-trading phases without reworking the data foundation.

Phase Goal

Deliver safe local storage, restart recovery, and a trustworthy decision/activity history without introducing execution capability.

Purpose

Phase 2 proved that VectorQuay can connect to Coinbase read-only and present truthful current-state data. Phase 3 must now make that state durable enough for operational continuity, debugging, later performance measurement, and future deterministic decisioning. The product must be able to restart cleanly, reconstruct recent truth, and explain how the current state was reached.

Carry-Forward Decisions

  • Coinbase Advanced Trade remains the only exchange surface.
  • Read-only exchange posture remains mandatory in Phase 3.
  • No live order placement, order cancellation, or paper trading is introduced in this phase.
  • Spot-only and trade-only scope remain intact.
  • Approved-candidate and protected-asset overlays remain locally authoritative.
  • USD remains the primary valuation currency, with USDC retained as an approved secondary quote/routing asset.

In Scope

  • Persist the last known successful Coinbase read-only snapshot needed for restart recovery and recent-history display.
  • Persist a bounded recent portfolio-value history sufficient for simple trend display, not a full market-data warehouse.
  • Persist audit records for refreshes, configuration changes, policy changes, alert-rule changes, and source/watcher changes.
  • Build a replayable local event trail showing what changed, when it changed, and why it changed.
  • Implement safe startup recovery that restores the last durable known state before the next refresh completes.
  • Improve Activity and Performance so they can use stored history rather than only current-session memory.
  • Preserve secrets outside the repository and outside the durable audit payloads.

Out of Scope

  • Live trading or paper trading.
  • Strategy scoring, risk approval, or execution simulation.
  • Historical backtesting and bulk replay tooling beyond what is needed for operational recovery and recent audit visibility.
  • Cloud sync, multi-machine replication, or remote storage.
  • AI-driven signals or watcher intelligence.

Persisted Data Classes

  • Last Known Account Snapshot: a cache-like restored-state record containing normalized balances, holdings summary, allocation summary, refresh timestamp, and connection-health result from the last successful live refresh.
  • Recent Activity History: bounded recent account/system events needed for cross-session Activity continuity.
  • Recent Alert History: bounded recent alert entries needed for cross-session alert visibility.
  • Recent Portfolio Value History: bounded timestamped portfolio-value points used only for simple trend/history display.
  • Audit/Event Trail: operator-trusted durable records for policy changes, alert-rule changes, source/watcher changes, configuration changes, startup recovery events, and refresh outcomes.

Retention Baseline

  • Recent Activity History should retain the latest 250 events by default.
  • Recent Alert History should retain the latest 250 alerts by default.
  • Recent Portfolio Value History should retain the latest 1,000 points by default.
  • Audit/Event Trail should remain append-oriented but rotate on a bounded size or count threshold rather than growing forever.

Primary Product Outcomes

  • The app can restart without losing track of the last known account and portfolio state.
  • The operator can inspect a durable history of important system and configuration events.
  • Activity and performance surfaces can start showing stored history instead of session-only state.
  • Future strategy, paper-trading, and live-validation phases can build on a stable persisted baseline.

Operator Problems Solved

  • The operator no longer loses useful app state when the shell closes or crashes.
  • The operator can see what changed in policies, alerts, and sources across sessions.
  • The operator can distinguish current-state truth from last-known persisted truth and new refresh results.
  • The operator can trust that later phases will have an audit trail instead of building one after execution logic exists.

Restored-State Semantics

  • The UI must distinguish three truth states explicitly: Live, Restored from Local Cache, and No Recoverable State.
  • If startup recovery succeeds before the next live refresh, restored state may be shown on Overview, Portfolio, Activity, Alerts, and Performance, but it must be labeled clearly as restored.
  • If the next live refresh succeeds, restored labels must be replaced by live-state cues.
  • If the next live refresh fails, restored state may remain visible as last-known truth, but the failure must be surfaced distinctly and must not be mistaken for current live data.
  • If no recoverable state exists, the shell must fall back to the existing non-connected / never-loaded behavior.

Screen-Level Expectations

  • Overview: should show last successful refresh time and whether current data is live or restored-from-cache.
  • Portfolio: should show last-known balance snapshot on startup before the next live refresh completes when recovery data exists.
  • Activity: becomes a durable local ledger of recent account events and system events rather than only current-session memory.
  • Performance: may start showing simple persisted trend information such as recent portfolio-value snapshots, but must not claim full backtested analytics.
  • Policies: edits become auditable, with durable change records.
  • Sources: local source/watcher changes become durable and auditable.
  • Alerts: alert-rule and delivery-preference changes become durable and auditable.
  • Configuration: can show storage/recovery health, but should remain primarily a connectivity/setup surface.

Phase 3 Functional Requirements

  • Persist the last known successful Coinbase read-only snapshot needed for safe startup recovery.
  • Persist only bounded recent histories; Phase 3 must not become an unbounded historical store.
  • Persist a local audit/event trail for operator changes and key system actions.
  • Restore last-known durable state on startup before or alongside the next approved live refresh attempt.
  • Ensure durable records are timestamped, typed, and explainable enough to reconstruct recent state changes.
  • Exclude secret values and secret material from persisted audit/event data.
  • Preserve the Phase 2 startup/manual refresh contract; persistence must not introduce silent polling.
  • Keep trading clearly inactive.

Success Criteria

  • The app restarts and shows last-known valid state safely even before a fresh live read completes.
  • Recent system/configuration changes are durable and inspectable.
  • The operator can reconstruct what happened across sessions without reading raw files.
  • No persisted data weakens the Phase 2 read-only or secret-handling boundary.

Phase 3 Is Complete When

  • Durable local storage exists for the approved Phase 3 data classes.
  • Startup recovery restores last-known state safely.
  • Recent event/audit history is visible and replayable enough for debugging and evaluation.
  • Phase 3 test plans pass and the product is ready for Phase 4 deterministic strategy/risk work.

Risks

Phase 3 Risks

  • Poorly scoped persistence could store secrets or unnecessary raw payloads.
  • Recovery logic could accidentally present stale state as current live truth without clear labeling.
  • Audit logging added too late or too weakly would leave later phases hard to debug or trust.
