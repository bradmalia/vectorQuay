Executive Summary

Phase 2 makes VectorQuay exchange-aware without enabling live order placement. It connects to Coinbase Advanced Trade in read-only mode, loads account balances and tradable products, shows connection health in the shell, and proves that the product understands available capital and Phase 0 restrictions before any execution path is considered.

Phase Goal

Deliver a reliable Coinbase read-only baseline that turns the Phase 1 shell into a truthful operator console for balances, products, prices, account state, and capital restrictions.

Purpose

Phase 1 created the product shell, local configuration model, and safety boundaries. Phase 2 must now connect those surfaces to real Coinbase data in a way that remains read-only, deterministic, and aligned with the approved v1 scope: Coinbase Advanced only, spot only, trade only, and existing-funds only.

Carry-Forward Decisions

  • Coinbase Advanced Trade remains the sole v1 exchange surface.
  • Spot-only and trade-only posture remain mandatory.
  • No bank funding, withdrawal, payment-method, external wallet, staking, Web3, margin, borrowing, or derivatives flows may appear as supported operator actions.
  • USD remains the preferred initial quote-currency and valuation path, with USDC preserved as an approved secondary routing asset.
  • Live trading remains restricted to a screened subset of assets; Phase 2 does not make every Coinbase-listed asset live-eligible.
  • No live order placement or cancellation may be implemented in this phase.

In Scope

  • Connect to Coinbase using read-only credentials loaded through the existing secret model.
  • Retrieve account balances, account state, tradable products, and current market prices needed for shell visibility.
  • Show connection health and last-refresh state in the GUI.
  • Populate Overview and Portfolio with real read-only Coinbase account data where available.
  • Populate Assets with real product-universe and approved-candidate status overlays.
  • Surface capital-rule visibility showing that only existing Coinbase funds are available for future trading.
  • Apply and display policy restrictions before any future execution path exists.

Out of Scope

  • Live order placement, order cancellation, or paper trading.
  • Historical data storage beyond what is strictly needed for current read-only shell operation.
  • AI-generated signals, watchers, or dynamic asset discovery.
  • Backtesting, simulation, strategy scoring, or portfolio optimization.
  • Any silent background automation that changes external exchange state.

Primary Product Outcomes

  • The shell can authenticate to Coinbase safely in read-only mode.
  • The operator can see accurate balances, products, prices, and connection state.
  • The product accurately distinguishes approved candidates, watch states, protected assets, and unavailable capital.
  • The shell remains visually clear that trading is still inactive even though read-only exchange data is live.
  • Phase 3 can begin storage and audit work without reworking the Coinbase integration boundary.

Primary Users and Stakeholders

  • Primary operator and owner: Brad Malia.
  • Future operator who needs truthful balance, product, and capital visibility before enabling any trading logic.
  • Future engineering work for persistence, audit, risk enforcement, and execution preparation.

Operator Problems Solved

  • The operator can verify whether Coinbase credentials are valid without risking any trade action.
  • The operator can see what the account actually holds instead of relying on mock shell data.
  • The operator can see which products are tradable, observed, screened, or restricted under Phase 0 decisions.
  • The operator can confirm the product understands existing capital and does not imply off-platform funding paths.

Non-Goals

  • Do not optimize returns in this phase.
  • Do not infer readiness for live trading just because account data is visible.
  • Do not broaden scope from Coinbase read-only into execution or automation.

Screen-Level Expectations

  • Overview: show real connection state, last refresh cues, real balances, and read-only account summary while preserving the inactive-trading message.
  • Configuration: show read-only connection setup status, credential-source status, release feed status, and non-secret settings actions.
  • Assets: replace mock rows with Coinbase product data plus approved/watch/restricted overlays from local policy state.
  • Policies: continue to show authoritative local policy state; do not imply that GUI edits bypass later enforcement.
  • Portfolio: show real holdings, quantities, valuation, allocation, and stable available-vs-held balance concepts where Coinbase exposes them.
  • Activity: remains a shell-oriented decision/activity surface in Phase 2 unless a simple read-only account-event view can be added without introducing persistence or misleading execution semantics. At minimum it must remain clearly inactive, consistent with current account truth, and must not imply that trade execution history is fully available.
  • Performance: remains a reserved shell surface in Phase 2 until persisted history exists. It must continue to render safely, avoid contradicting current account values, and explicitly communicate that historical performance metrics are not yet authoritative.
  • Risk & Thresholds: remain local/operator-safe in Phase 2; no automated enforcement actions yet.
  • Sources: Coinbase becomes the primary real direct source; watcher flows remain later-phase placeholders.
  • Alerts: may surface connection and validation issues, but not autonomous trading alerts.
  • About: continue to show version, update-check, and product metadata consistently.

Phase 2 Functional Requirements

  • Support read-only Coinbase authentication through the existing secret-file and environment-variable contract, using Coinbase credentials provisioned with read-only permissions only.
  • Use the exact Coinbase secret names VECTORQUAY_COINBASE_API_KEY and VECTORQUAY_COINBASE_API_SECRET for Phase 2 read-only authentication.
  • Retrieve account balances and current portfolio state through one clear refresh posture: attempt a safe startup refresh when a complete Coinbase credential set is present, and always support operator-triggered manual refresh.
  • Retrieve product metadata needed to determine which Coinbase spot products are available for analysis and future trading.
  • Retrieve market-price data required for truthful current-value display.
  • Compute and display available capital using only existing Coinbase balances.
  • Preserve the approved candidate class and DOGE watchlist posture in the GUI despite broader product visibility.
  • Preserve local policy overlays as authoritative operator state; Coinbase refreshes must not overwrite, remove, or silently reinterpret local policy restrictions.
  • Show clear failure states for missing credentials, rejected authentication, API unavailability, partial data, and incompatible account/product conditions.

Success Criteria

  • The operator can launch the shell, connect read-only to Coinbase, and see truthful account state without any trade action being possible.
  • Balances and product data are accurate enough to support later decisioning and restriction work.
  • The product continues to present an inactive trading state and does not imply that execution is ready.
  • The approved Phase 0 and Phase 1 constraints remain visible in the product, not only in the documents.

Phase 2 Is Complete When

  • Coinbase read-only integration works through the approved secret model.
  • The startup refresh posture and manual refresh behavior are implemented exactly as approved, with no silent background polling.
  • Overview, Assets, Portfolio, Configuration, and connection-health surfaces present truthful read-only data.
  • The system correctly distinguishes existing funds from unsupported funding paths.
  • Policy restrictions and approved-candidate overlays are visible before any execution path exists.
  • Phase 2 test plans pass and the product is ready for Phase 3 storage and audit work.

Risks

Phase 2 Risks

  • Misreading Coinbase balances or product flags would contaminate later execution logic.
  • Weak error handling around credentials or partial API responses would make the shell look more reliable than it is.
  • Allowing read-only connectivity to blur into execution semantics would create unsafe scope drift.
