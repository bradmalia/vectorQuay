Phase 2 Technical Requirements

Read-only Coinbase integration, shell data binding, and capital-rule interpretation requirements.

Versionv0.1
StatusDraft baseline
OwnerBrad Malia
ThemeIndigo Signal

Executive Summary

Phase 2 technical work should introduce a read-only Coinbase integration layer that fits cleanly into the existing Phase 1 solution structure. The shell should start consuming truthful exchange data while preserving deterministic local settings behavior and a hard no-execution boundary.

Technical Goal

Make Coinbase data available to the shell without letting exchange concerns leak across the entire app or imply live trade capability.

Implementation Boundaries

  • Continue using Avalonia on .NET 8 and the existing `src/VectorQuay.App`, `src/VectorQuay.Core`, and `tests/` solution shape.
  • Add Coinbase-specific code in a separate integration-focused area rather than crowding view models with HTTP and signature logic.
  • Preserve the existing secret-file contract at ${XDG_CONFIG_HOME:-~/.config}/VectorQuay/secrets.env with environment-variable override precedence.
  • Use an explicit Coinbase credential contract. Phase 2 must require VECTORQUAY_COINBASE_API_KEY and VECTORQUAY_COINBASE_API_SECRET before the shell attempts authenticated refresh.
  • Do not add any write-capable Coinbase operations in Phase 2.

Required Technical Capabilities

  • Read-only Coinbase client capable of authenticated requests for account balances and tradable products.
  • Market-data retrieval sufficient for current valuation and product visibility.
  • A mapping layer that converts Coinbase responses into stable application models for Overview, Portfolio, Assets, and Configuration.
  • Deterministic failure classification for missing credentials, auth failures, network failures, rate limits, and partial-response conditions.
  • Explicit connection-health state with last successful refresh time and current error state.

Credential Contract

  • The required Coinbase secret names are VECTORQUAY_COINBASE_API_KEY and VECTORQUAY_COINBASE_API_SECRET.
  • The Coinbase API key used for Phase 2 must be provisioned with read-only permissions only.
  • The shell must classify credential state as missing, incomplete, or present without exposing secret values.
  • If the required Coinbase credential set is incomplete, the app must not attempt authenticated exchange refresh.
  • Credential failures must produce stable operator-facing error summaries and developer-readable diagnostics.

Architecture Requirements

  • Define Coinbase DTOs separately from view models.
  • Define app-facing read models separately from raw Coinbase payloads.
  • Keep signature/auth logic isolated and testable.
  • Keep view models dependent on app-facing models and services, not direct HTTP calls.

Data Refresh Requirements

  • Support manual refresh from the shell.
  • Support one approved startup posture: attempt a safe startup refresh once when a complete Coinbase credential set is present.
  • Do not introduce silent background polling as part of the initial Phase 2 baseline unless a later approved revision defines cadence, rate-limit handling, and stale-state semantics explicitly.
  • Define stale, unavailable, never-loaded, and failed-refresh states distinctly in app-facing models.
  • Define rate-limit and retry behavior explicitly before implementation; default behavior should favor operator-visible failure over aggressive retry loops.
  • Failures must not crash the shell.
  • Current display must distinguish stale, unavailable, and never-loaded data.

Screen Interpretation Requirements

  • Overview: bind current account summary, connection health, and refresh cues to real data.
  • Configuration: show credential-source status, connection result, valuation-currency lock, and read-only exchange setup interpretation.
  • Assets: merge Coinbase product list with local approved/watch/restricted overlays.
  • Portfolio: present holdings, valuation, and allocation from Coinbase read-only data without implying historical accounting beyond current state.
  • Sources: show Coinbase as an active direct source; watcher rows remain shell placeholders.

Capital and Restriction Rules

  • The implementation must derive available capital only from existing Coinbase balances.
  • The shell must not imply off-platform deposits, bank funding, or external-wallet transfers as normal funding paths.
  • USD remains the locked valuation currency in the app-facing models.
  • USDC may appear in app-facing models as an approved secondary quote/routing asset but must not replace USD as the default interpretation.
  • Approved-candidate overlays and protected-asset policies remain locally authoritative and must be visible alongside Coinbase product data.
  • Exchange refresh must not overwrite or silently reinterpret local policy overlays.

Persistence Boundary

  • Phase 2 must not introduce durable storage of Coinbase payloads, derived market state, or account-history snapshots as a normal application behavior.
  • Read-only exchange data should remain in-memory and session-scoped unless a later approved phase explicitly adds persistence.
  • Existing local settings persistence remains allowed only for the Phase 1 non-secret configuration and policy contract.

Logging and Diagnostics

  • Log connection attempts and failures in a developer-readable way without exposing secret values.
  • Surface concise operator-facing error summaries in the shell.
  • Keep the app runnable even when Coinbase is unreachable or credentials are invalid.

Delivery Requirements

  • Build and run on the existing Ubuntu-first local environment with .NET 8.
  • Preserve Windows compatibility in the project structure and packaging assets.
  • Add or update tests under tests/ for new Coinbase-facing logic and mapping rules.
  • Record Phase 2 validation artifacts under docs/test-results/ when execution begins.

Phase 2 Technical Completion Criteria

  • Read-only Coinbase services are isolated, testable, and wired into the shell cleanly.
  • The exact Coinbase credential contract and approved startup/manual refresh behavior are implemented without background polling.
  • Connection health, balances, products, and capital-rule interpretation work without crashing the UI.
  • No execution path exists.
  • The Phase 1 configuration and secret-handling guarantees remain intact.
