Technical Requirements Document

Phase 4 Technical Requirements

Deterministic strategy pipeline, risk approval engine, and auditable decision modeling.

Versionv0.1
StatusDraft baseline
OwnerBrad Malia
ThemeIndigo Signal

Executive Summary

Phase 4 technical work adds the first deterministic strategy and risk engine on top of the Phase 3 durable-state foundation. The output should be a testable decision pipeline that can score trade candidates, enforce hard constraints, and persist/replay its decisions without introducing execution behavior.

Technical Goal

Separate signal, scoring, and risk approval cleanly enough that later backtesting and paper trading can reuse the same pipeline.

Implementation Boundaries

  • Preserve the existing .NET 8 / Avalonia / Core/App solution structure.
  • Add strategy and risk logic as isolated services/models in shared/core code, not directly inside the view models.
  • Use current Coinbase read-only state and current local policy/risk state as inputs only.
  • Do not introduce live order submission, simulated execution, or paper-order lifecycle in Phase 4.
  • Preserve the durable-state/audit contract introduced in Phase 3.

Required Technical Capabilities

  • Typed candidate-generation models using current normalized account/product/state inputs.
  • Typed scoring models with deterministic component weights and rationale fields.
  • Typed risk-approval results with explicit rejection/constraint reasons.
  • Durable recording of strategy decisions into the existing Phase 3 event/audit layer.
  • Profile-driven threshold configuration that is deterministic and versionable.

Scoring Contract

  • Phase 4 must define one explicit normalized score range: 0.0 to 1.0.
  • The final score must be composed from named deterministic components, each with an explicit bounded range and documented weight.
  • The initial component set should include at minimum:
    • trend/momentum strength
    • liquidity / spread quality
    • portfolio cash availability
    • concentration penalty
    • turnover/churn penalty
  • The default component weights should be:
    • trend/momentum strength: 0.35
    • liquidity / spread quality: 0.20
    • portfolio cash availability: 0.15
    • concentration penalty: 0.15
    • turnover/churn penalty: 0.15
  • Weights may differ by aggressiveness profile, but the weights and ranges must remain fixed and inspectable within a given build/runtime configuration.
  • The persisted decision record must include both the final score and the component-level breakdown used to produce it.

Decision Pipeline Contract

  • Universe Eligibility Pass: evaluate whether an asset is allowed to enter the approved trade universe using exchange availability, quote-path support, liquidity/spread thresholds, and operator policy constraints.
  • Input Stage: consume a normalized snapshot of current balances, holdings, product data, approved candidates, protected-asset policies, and active risk profile.
  • Signal Stage: calculate deterministic strategy inputs only from currently approved internal data sources.
  • Scoring Stage: produce a normalized candidate score and component explanations.
  • Risk Stage: apply non-bypassable hard constraints after scoring and before final approval output.
  • Output Stage: persist a typed decision result containing status, reasons, candidate size suggestion if applicable, and all relevant blocking/constraint signals.

Eligibility and Reevaluation Requirements

  • The eligibility contract must be evaluated separately from trade approval.
  • The default eligibility thresholds should be:
    • minimum 24H liquidity: $250,000
    • maximum quoted spread tolerance: 0.40%
  • Eligibility must run at minimum after a successful startup/live refresh, after an explicit Refresh Universe action, and after approved-universe or policy-state changes.
  • Trade-candidate evaluation must run after each successful live refresh and after relevant policy, profile, or threshold changes.
  • Risk-profile changes must affect trade approval and scoring sensitivity, not silently reclassify universe membership unless a hard protected-asset rule fully blocks trading.
  • The default execution model should be:
    • full-universe eligibility reevaluation after startup refresh and explicit Refresh Universe
    • full-approved-subset decision reevaluation after successful live refresh
    • targeted reevaluation of only affected assets after a single-asset policy change where technically practical; otherwise fall back safely to full-approved-subset reevaluation
  • Reevaluation may be synchronous in Phase 4 if the bounded asset set keeps latency acceptable; asynchronous/background reevaluation should not be introduced unless explicitly approved later.
  • The default evaluation shape should be independent per asset, then ranked. Assets are first evaluated independently, then ranked for operator-facing candidate summaries.

Evaluation State and Progress Requirements

  • The decision engine must expose a typed operator-facing evaluation state model rather than leaving reevaluation silent.
  • The minimum evaluation states are: Idle, RefreshingData, EvaluatingUniverse, ScoringCandidates, ApplyingRiskChecks, Completed, and Failed.
  • The app-facing layer must surface the current evaluation state to the footer/status bar and to operator-triggered actions such as Refresh Universe.
  • On successful completion, the app-facing layer must expose a durable last-evaluated timestamp.
  • On failure, the app-facing layer must expose which evaluation stage failed and preserve the last known decision state safely.

Hard Risk Contract

  • Hard limits must be evaluated independently from candidate attractiveness.
  • At minimum, enforce checks for:
    • asset eligibility / approved-universe membership
    • protected-asset mode
    • maximum position size
    • maximum concentration / exposure
    • daily loss budget
    • turnover budget
    • stale data
    • refresh / reconciliation / connectivity health
    • operator kill-switch / inactive-trading state
  • Risk rejection reasons must be explicit, typed, and durable.

Candidate Sizing Contract

  • Phase 4 must produce a deterministic suggested size for Approved or Constrained decisions even though no order is placed.
  • The initial sizing model should be derived from available deployable cash, capped by:
    • maximum position size
    • maximum concentration / exposure
    • minimum trade threshold
    • protected-asset and turnover constraints
  • The sizing result must include a size rationale or cap reason whenever the raw desired size is reduced or zeroed.
  • The default minimum trade threshold should be explicit and operator-visible; the baseline Phase 4 default should be $10.00 notional.

Decision Explainability Contract

  • Each decision output must include separate explanation sections for:
    • eligibility result
    • score components and rationale
    • hard-risk checks and blocking reasons
    • final disposition and any size cap or no-action reason
  • Explainability records must be durable and distinguishable from true account-event history.
  • The UI-facing model must be explicit enough that an operator can validate whether the system made the right decision without reading raw files.

Profile Configuration Requirements

  • Profiles must be explicit, named, and deterministic: Conservative, Balanced, Aggressive at minimum.
  • Profiles may adjust soft strategy thresholds and ranking sensitivity.
  • Profiles must not modify or bypass hard risk rules.
  • Profile changes must be persisted and auditable.

Decision Persistence Requirements

  • Each candidate decision record must include: decision schema version, decision id, timestamp, asset, proposed action, candidate score, score-component breakdown, final status, rationale summary, blocking/constraint reasons, profile/risk context, and input snapshot correlation id.
  • Decision records must be durable enough for Activity and future backtesting/replay inspection.
  • Decision records must remain separate from true exchange-account events so the UI can distinguish hypothetical strategy decisions from actual account history.
  • Persistent No Action decisions should be stored selectively rather than exhaustively, so the durable log remains operator-meaningful.
  • The default operator-facing retention model should persist all Approved, Constrained, and operator-relevant Rejected decisions, then surface only the top ranked subset in compact summary views such as Overview.

Testing Requirements

  • Add automated tests for universe-eligibility rules and reevaluation timing triggers where logic is isolated enough to unit test.
  • Add automated tests for deterministic scoring repeatability.
  • Add automated tests for hard-risk rule enforcement and non-bypassability.
  • Add automated tests that prove profile changes affect soft thresholds but not hard safety rules.
  • Add automated tests for decision explainability payload completeness.
  • Add automated tests for decision persistence and replayability.
  • Add integration/regression validation that the app still does not place orders or imply execution readiness.

Phase 4 Technical Completion Criteria

  • Strategy, scoring, and risk-approval services are isolated and testable.
  • Decision outputs are durable and inspectable.
  • Risk rules are demonstrably non-bypassable.
  • No execution path exists.
  • The decision pipeline is stable enough to become the baseline for Phase 5 backtesting.
