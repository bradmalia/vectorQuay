Phase 3 Technical Requirements

Local persistence, event/audit modeling, and restart recovery requirements.

Versionv0.1
StatusDraft baseline
OwnerBrad Malia
ThemeIndigo Signal

Executive Summary

Phase 3 technical work adds a local persistence layer and a replayable event trail without breaking the existing read-only exchange boundary. The output should be a restart-safe, auditable shell that preserves operational continuity and gives later phases a durable foundation for strategy and paper-trading work.

Technical Goal

Persist only what the product needs, label restored state truthfully, and make operator/system changes reconstructable.

Implementation Boundaries

  • Preserve the existing .NET 8 / Avalonia / Core/App solution shape.
  • Add storage and audit logic in isolated services/models rather than directly in view models.
  • Keep Coinbase integration read-only.
  • Do not persist secret values, secret files, or secret-derived plaintext into the new storage layer.
  • Do not introduce background polling or recovery behaviors that mutate exchange state.

Required Technical Capabilities

  • Durable local storage for approved persisted models.
  • Typed event/audit records for operator changes and key system actions.
  • Startup recovery service that can load last-known durable state before the next refresh.
  • Versioned or forward-compatible persisted document/schema handling.
  • Safe corruption handling with graceful fallback when durable files are unreadable.

Persistence Boundary

  • Persisted Phase 3 data may include: last-known Coinbase summary state, bounded recent account-event history, bounded recent alert history, source/watcher registry state, policy state, bounded recent portfolio-value history, and audit/event records for local changes.
  • Persisted Phase 3 data must not include: secret values, OAuth tokens, API secrets, raw provider credentials, or unnecessary raw exchange payload dumps.
  • Where possible, store normalized app-facing models instead of entire raw provider responses.
  • Storage should remain local to the current machine and operator profile.

Persisted Model Contract

  • The minimum persisted snapshot contract must include: snapshot timestamp, refresh outcome, connection-health classification, normalized balances summary, holdings summary, and allocation summary.
  • Persisted snapshot data is cache-like restored state, not an audit-grade record of truth by itself.
  • The minimum persisted recent-activity contract must include: event timestamp, event type, asset or subject, status, summary/detail text, and durable source classification.
  • The minimum persisted portfolio-history contract must include: timestamp and total portfolio value in USD.
  • Persisted product visibility should be summarized only as needed for restart UX; Phase 3 must not store the full Coinbase product universe unless later approved explicitly.
  • Audit/event records are operator-trusted records and must be modeled separately from cache-like restored snapshots.

Recovery Requirements

  • Startup should load the last known durable state before or while a new approved refresh is attempted.
  • The UI must distinguish restored cached state from newly refreshed live state using explicit app-facing status values.
  • The minimum recovery truth states are Live, RestoredFromCache, and NoRecoverableState.
  • If live refresh fails after recovery succeeds, the restored state may remain visible, but the shell must show the refresh failure distinctly and preserve the restored-state label.
  • If storage is missing or corrupted, the app must fall back safely without crashing.
  • Recovery must not trigger repeated background retry loops.

Audit Requirements

  • Every persisted audit/event record must include at minimum: event id, timestamp, event type, origin, summary, and redacted detail payload.
  • Where a durable change modifies prior state, the record should include before/after values or a normalized diff sufficient to reconstruct the change.
  • Refresh-related records should include a correlation identifier or session identifier so startup/recovery/refresh events can be grouped.
  • Operator-originated changes should record origin like policy edit, alert-rule save, source add/remove, and configuration update.
  • System-originated changes should record origin like startup recovery loaded, Coinbase refresh succeeded, Coinbase refresh failed, or icon cache updated.
  • Audit records must remain readable without inspecting raw JSON manually.

Redaction Rules

  • Secret values, secret file contents, raw credential strings, authorization headers, and provider tokens must never be persisted.
  • Connection test messages and refresh errors may be persisted only in redacted/operator-safe form.
  • Free-text operator notes, source prompts, and alert destinations may be persisted only if they are treated as operator-authored local configuration data and not mixed with secret-bearing diagnostic payloads.
  • Where free-text fields may contain risky values, the storage layer should trim or redact them rather than persist raw content automatically.

Storage Design Requirements

  • Use a deterministic local storage location under the approved user config/app-data path rather than inside the repository.
  • Use explicit file names or subdirectories for snapshots, audit/event logs, and schema/version markers.
  • The baseline local storage layout should be defined explicitly before implementation, for example: ~/.config/VectorQuay/data/state/last-snapshot.json, ~/.config/VectorQuay/data/history/activity.jsonl, ~/.config/VectorQuay/data/history/alerts.jsonl, ~/.config/VectorQuay/data/history/portfolio-value.jsonl, and ~/.config/VectorQuay/data/audit/events.jsonl.
  • The storage layer should include a schema/version marker so future migration logic has a stable anchor.
  • Writes should be atomic enough to avoid partially written state on interruption.
  • Retention should be bounded where needed so local logs do not grow without limit.
  • The baseline retention contract is: 250 recent activity events, 250 recent alerts, 1,000 recent portfolio-value points, and audit-log rotation on a bounded size or count threshold.

Screen Interpretation Requirements

  • Overview: may show last live refresh and restored-state cues from the persistence layer.
  • Portfolio: may use persisted last-known balances during startup recovery.
  • Activity: should unify recent durable account/system events into a replayable ledger.
  • Performance: may use persisted historical snapshots to show simple value trends, but must not claim full strategy analytics.
  • Configuration: may show storage/recovery health or audit-file status only if it improves operator clarity without becoming noisy.

Testing Requirements

  • Add automated tests for serialization, normalization, migration/compatibility handling, and corruption fallback.
  • Add automated tests for startup recovery behavior and for preserving the read-only boundary.
  • Add automated tests for audit/event recording of policy, alert, and source changes.
  • Add automated tests for restored-state labeling rules so cached state is never presented as fresh live state incorrectly.
  • Update validation artifacts under docs/test-results/ for Phase 3 execution.

Phase 3 Technical Completion Criteria

  • Durable state and audit/event services are isolated and testable.
  • Startup recovery works without weakening the Phase 2 refresh contract.
  • Persisted state is safe, local, and excludes secrets.
  • Activity and recent-history surfaces can consume durable data.
  • No execution path exists.
