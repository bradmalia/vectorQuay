Phase 1 Technical Requirements

Repository, application-shell, local settings, and security baseline requirements for the first implementation phase.

StatusApproved

Executive Summary

Phase 1 technical work turns the approved Phase 1 PRD into a concrete implementation baseline. It defines the repository structure, Avalonia application shape, local settings model, secret-handling boundaries, shell-view architecture, and delivery requirements needed before Phase 2 Coinbase read-only integration begins.

Primary Output

A runnable Ubuntu-first .NET/Avalonia application shell with deterministic local configuration and safe persistence boundaries, ready for Phase 2 exchange integration.

1. Scope

  • Translate the approved Phase 1 PRD into concrete technical implementation requirements.
  • Define the repository and solution structure for the first production-quality baseline.
  • Define the application-shell architecture for the Avalonia GUI.
  • Define local settings, secret loading, configuration precedence, and ignored-file boundaries.
  • Define the technical contract for the reserved Phase 1 screens without over-implementing later-phase trading behavior.
  • Define the build, run, validation, logging, and documentation expectations required to close Phase 1 cleanly.

2. Out of Scope

  • Live Coinbase order placement or cancellation.
  • Paper trading.
  • Production strategy logic, signal scoring, or portfolio optimization.
  • AI-driven trading, news analysis, or live source ingestion.
  • Background asset-discovery jobs beyond shell reservation and placeholder refresh affordances.
  • Automated self-update installation.
  • Container-first packaging or deployment automation.

3. Technical Objectives

  • Create a maintainable private-repository baseline that can support later resale and credential isolation.
  • Establish Avalonia UI on .NET 8 as the implementation baseline for Ubuntu-first local operation with a Windows 11 compatibility path.
  • Define a deterministic local configuration model with app-managed non-secret settings and external secrets.
  • Define shell navigation, view models, and placeholder states for the approved Phase 1 screens.
  • Preserve the approved product boundaries from Phase 0, including screened live trading universe rules and future dynamic asset discovery.
  • Keep later-phase features visible in the shell without implying they are operational in Phase 1.
  • Produce a technical baseline that can accept Phase 2 Coinbase read-only integration without reworking core application structure.

4. Technical Deliverables

  • Phase 1 technical requirements document.
  • Private GitHub repository with approved Phase 0 and Phase 1 document baseline committed.
  • .NET solution and project scaffold for the GUI shell and shared application foundation.
  • Runnable Avalonia shell application.
  • Configuration templates, ignored local settings path, and ignored secret-file path.
  • Documented build-and-run workflow for Ubuntu primary development.
  • Updated context and decision-log artifacts for future sessions.
  • Phase 1 unit, integration, and regression test plans.

5. Repository and Source-Control Requirements

  • Phase 1 must create the private GitHub repository as the canonical source-control home for VectorQuay.
  • The repository must contain the approved Phase 0 documentation baseline as the first project record.
  • The repository must establish a coherent top-level structure for source, tests, docs, assets, and configuration templates.
  • The default branch must be documented and used consistently for the baseline implementation path.
  • The ignored-files policy must explicitly exclude local secrets, local app-managed settings, machine-specific artifacts, and generated outputs not intended for version control.
  • The repository must remain compatible with future resale and BYO-credentials distribution rather than assuming one operator or one machine forever.

6. Solution and Application Architecture

Solution Baseline

  • Create a primary .NET solution targeting .NET 8.
  • Separate the GUI shell from shared domain/configuration logic so Phase 2 integration work does not crowd the UI layer.
  • Preserve room for future test projects, service adapters, and data-access layers without flattening everything into the GUI project.
  • Use naming that matches product language already approved in the PRD.

GUI Stack

  • Use Avalonia UI on .NET 8 as the default Phase 1 implementation stack.
  • Do not require Docker for build, run, or local configuration.
  • Preserve a Windows-compatible project shape while treating Ubuntu as the primary environment.
  • Reserve later packaging choices rather than overcommitting to self-updating installers in Phase 1.

6A. Proposed Project Structure

  • Phase 1 should establish a solution shape equivalent to: src/VectorQuay.App for the Avalonia shell, src/VectorQuay.Core for shared domain and settings logic, and one or more future-ready test projects under tests/.
  • The GUI shell project should not absorb all configuration, domain, and future integration logic directly.
  • Documentation, templates, assets, and context artifacts should remain outside the application project tree in stable repository locations.
  • The actual file and project names may vary slightly, but the separation of shell, shared logic, tests, docs, and templates must be preserved.

7. Shell Navigation and View Architecture

  • The shell must implement the approved top-level navigation consistently across the application.
  • The approved Phase 1 PRD mock screens remain the visual and structural UI reference for the shell; this TRD defines the technical interpretation, state behavior, persistence rules, and implementation boundaries for those screens.
  • The Phase 1 shell must include the approved views: Home, Configuration, Assets, Policies, Portfolio, Activity, Performance, Risk & Thresholds, Sources, Alerts, and About.
  • Every approved top-level screen must have a stable view model and placeholder rendering path, even where real data is deferred.
  • Top-level navigation state must be centralized rather than reimplemented separately inside each view.
  • Reserved later-phase behavior must be communicated as inactive, placeholder, or future-ready rather than silently omitted.
  • The shell must support clear “not configured”, “inactive”, and “not connected” states without implying live trading or paper trading readiness.
  • The Phase 1 shell must not expose or imply supported operator flows for bank funding, bank withdrawal, payment-method management, external wallet transfer, staking, Web3/on-chain actions, margin, borrowing, derivatives, or other non-trade-only account actions.
  • Branding, version, and owner attribution must appear consistently across the shell through the application title/window metadata, shared navigation or header chrome, the Home status surface, and the About screen.

7A. UI State and Interaction Model

  • Phase 1 should use an MVVM-style separation between views, view models, and shared application services.
  • Navigation state, settings state, validation state, and placeholder shell state should be centralized rather than embedded directly in view code.
  • Editable controls must distinguish clearly between: read-only derived values, editable non-secret operator settings, and unavailable future functionality.
  • Meaningful operator actions such as Save, Validate, Reset to Default, and profile switching should route through explicit command handlers rather than ad hoc code-behind mutations.

8. Screen-by-Screen Technical Requirements

Home / Status

  • Render a dashboard view with application state, connection state placeholders, recent activity placeholders, alerts area, and top candidate placeholders.
  • Display software version, last-updated cues, and inactive-trading state clearly.
  • Do not depend on live Coinbase data in Phase 1.

Configuration

  • Render editable non-secret settings and secret-source status without exposing actual secret values.
  • Show non-editable values as disabled or greyed-out, including detected platform and the locked Phase 1 valuation currency of USD.
  • Phase 1 must treat USD as the preferred initial quote-currency and routing path for the current production-account posture, while preserving USDC as an approved secondary quote/routing asset for later enforcement and integration work.
  • Support save and validate actions against deterministic local settings rules.

Assets

  • Show the approved initial candidate class and DOGE conditional watchlist note.
  • Support filter placeholders for asset and state plus a default rank/order by trade priority.
  • Reserve manual refresh UI and future automatic discovery state without implementing discovery jobs in Phase 1.

Policies

  • Support operator-safe local editing of approved candidate notes, protected-asset modes, and related non-secret policy state.
  • Provide row selection, checkbox-based multi-select or equivalent row-state clarity, filters, and selected-policy details in the shell.
  • Use clear operator-facing labels such as Do Not Buy, Do Not Sell, and Do Not Trade or their approved equivalents.
  • Phase 1 policy edits must persist as authoritative local operator policy state, but the shell must make clear that GUI edits do not bypass later technical enforcement boundaries.

Portfolio and Performance

  • Reserve portfolio holdings, allocation, KPI, and chart areas as placeholder UI sections.
  • Reserve selectable time ranges of 24H, 7D, 30D, 90D, 1Y, and ALL for performance without implying live metric calculation in Phase 1.
  • Retain standard metric names such as PnL, Win Rate, and Drawdown while supporting tooltip or inline-help affordances later.

Activity

  • Implement the shell shape for a single decision-led activity view rather than separate trade-history and decision-trace screens.
  • Reserve filtering and sorting by timestamp/date range, asset, action, source, status, and decision outcome.
  • Support selected-row detail state, policy/risk check placeholders, and source-contribution placeholders without implementing real decision logic yet.

Risk & Thresholds

  • Support the approved High Risk, Medium Risk, Low Risk, and Custom profile model in shell form.
  • Preset profiles must appear as recommended defaults; Custom must be the only editing mode for direct threshold value changes.
  • Switching from Custom to a preset must require confirmation when unsaved custom changes would be overwritten.
  • Reset to Default must restore the recommended/default state explicitly.

Sources

  • Implement the unified source registry shell with tabs for All, Direct Sources, and Watchers.
  • Reserve filter controls, checkbox-based row selection or equivalent multi-select clarity, and selected-item details for direct sources and watchers in one view.
  • The reserved source state model must be locked in Phase 1 as Active, Observed, Ignored, Blocked, and Needs Review.
  • Support placeholder actions for weight editing, scope editing, history, automation settings, reliability review, and grouped state actions.
  • The technical model must reserve manual direct-source addition by URL or handle plus AI-assisted watcher creation from prompts, with both represented in the same registry structure.
  • AI-created watchers must enter the registry in Observed or Needs Review state by default until later-phase logic refines their lifecycle.
  • Keep ignore and block reversible; reactivate/unblock and reset-to-default must remain part of the reserved technical model.

Alerts

  • Reserve recent-alert history plus notification preference controls for categories, severities, channels, and quiet hours.
  • Do not implement outbound email or SMS delivery in Phase 1.

About

  • Show version, attribution, documentation references, and configured release-source update controls in shell form.
  • Support an owner-configurable manual tip section that is clearly disconnected from bot trading behavior and may be disabled in resale builds.

9. Configuration, Settings, and Secrets

Source of Truth

  • Non-secret operator settings and policy edits must persist through application-managed local settings.
  • Checked-in files must remain templates only.
  • Secrets must remain external to source control.
  • Configuration precedence must be deterministic and documented in code and docs.
  • The Phase 1 baseline must use this precedence order unless an approved later decision changes it: application defaults, checked-in templates, application-managed local non-secret settings, external secret file values, and environment variables as the highest-precedence secret override.
  • The application must be able to start safely when secrets are missing, while clearly surfacing the missing state to the operator.
  • Secret values must never be rendered in plain text in normal UI states.
  • Logs and validation errors must redact or avoid secret values.
  • Local override paths and ignored files must be documented and enforced through source-control ignore rules.
  • The technical baseline must preserve BYO-credentials support for future resale and multi-user distribution.
  • The checked-in template for non-secret configuration must live in a stable repository path such as config/templates/appsettings.template.json and must never be treated as the live operator settings file.

9A. Local Storage Paths

  • For Ubuntu-first operation, the application-managed non-secret settings file must be stored at ${XDG_CONFIG_HOME:-~/.config}/VectorQuay/settings.json.
  • For Ubuntu-first operation, the external secret file must be stored at ${XDG_CONFIG_HOME:-~/.config}/VectorQuay/secrets.env.
  • The non-secret settings file format must be JSON, and the external secret file format must be simple key/value environment-style entries.
  • The Phase 1 secret contract must reserve at least VECTORQUAY_COINBASE_API_KEY, VECTORQUAY_COINBASE_API_SECRET, and VECTORQUAY_OPENAI_API_KEY as supported secret names.
  • Environment variables with the same names must override values supplied through the external secret file.
  • The application should surface the existence and health of those sources in the UI without exposing raw secret values.
  • For Windows compatibility, the design must map the same settings model onto the local application-data equivalent rather than hard-coding Ubuntu-specific path logic into business or UI code.

10. Build, Run, and Validation Requirements

  • Phase 1 must define one clear documented workflow to restore, build, and run the application on Ubuntu.
  • The baseline application must launch without live Coinbase connectivity.
  • Validation commands and expected operator setup steps must be documented in the repository.
  • The application must expose understandable startup and validation failures rather than crashing without explanation.
  • The build and run path must not require Docker.

10A. Validation Contract

  • Validation must cover at least: settings file readability, secret-source presence detection, shell startup viability, and placeholder-view navigation integrity.
  • Validation results should distinguish between blocking failures, warnings, and informational inactive states.
  • The Phase 1 shell should be able to report “not configured yet” as a valid state rather than treating it as a crash condition.
  • The documented validation flow must verify at minimum that: the application launches successfully with no secrets present, every approved top-level view can be opened, non-secret policy and configuration edits persist across restart, logs and validation output do not reveal secret values, and local ignored-file rules cover the application-managed settings file and external secret file.

11. Logging, Errors, and Operator Safety

  • Phase 1 logging must support troubleshooting of config, startup, and shell-state issues without exposing secrets.
  • Validation and startup errors must be understandable in operator-facing language where appropriate.
  • The shell must clearly distinguish between inactive placeholders and genuinely configured functionality.
  • No screen should imply that live trading, paper trading, or live source ingestion is active in Phase 1.

12. Documentation and Context Preservation

  • Phase 1 must preserve roadmap, Phase 0 closeout, PRD, and TRD alignment in-repo.
  • Implementation decisions made during Phase 1 must be stored in durable project context or decision-log artifacts.
  • The actual private GitHub repository URL should be recorded in project context or technical metadata after repository creation, not embedded as a product requirement.
  • The repository must remain understandable without dependence on prior chat history.

13. Technical Decision Outputs Required

  • What exact solution/project structure will be used for the Avalonia shell and shared application logic?
  • The application-managed non-secret settings contract is locked as JSON at ${XDG_CONFIG_HOME:-~/.config}/VectorQuay/settings.json.
  • The ignored secret-file contract is locked as ${XDG_CONFIG_HOME:-~/.config}/VectorQuay/secrets.env with environment-variable override support for future Coinbase and OpenAI credentials.
  • What build and run commands will be the standard Ubuntu development workflow?
  • What validation flow will confirm that configuration, shell startup, and placeholder navigation are working?
  • What project context files will be updated as the implementation baseline is created?
  • The source-registry state contract is locked as Active, Observed, Ignored, Blocked, and Needs Review.
  • The performance-range contract is locked as 24H, 7D, 30D, 90D, 1Y, and ALL.

14. Security and Secrets

  • No real secrets may be committed to the repository, screenshots, sample settings, or logs.
  • Only non-secret templates may be checked in.
  • Secret-source status may be shown in the GUI, but the secret values themselves must not be exposed in routine UI or logs.
  • Owner-configurable features such as the tip section must never gain implicit access to exchange funds or bot execution paths.
  • Update checking must target a configured release source and must not assume a private GitHub-only consumer path for future resale builds.

14A. Update and Release Boundary

  • Phase 1 must implement a real manual update check in the About view against GitHub Releases as the initial configured release source, showing current version, latest available version, and user-facing download/open actions when an update is available.
  • Any update-check implementation path introduced later must depend on a configurable release-source abstraction rather than private-repository assumptions embedded in UI logic.
  • The Phase 1 update flow must remain manual-only and must not perform silent self-updating, background installation, or forced restart behavior.
  • Version display should use a coherent application version contract across screens and docs.

15. Non-Functional Technical Requirements

  • The codebase must remain maintainable and testable rather than shortcut prototype code.
  • The shell must remain visually consistent with the approved branding and document mocks.
  • The technical baseline must be understandable to future sessions without chat dependence.
  • The design must preserve later expansion for Coinbase integration, dynamic asset discovery, source intelligence, alerts, and release management.

16. Entry and Exit Criteria

Entry Criteria

  • Approved roadmap v4 exists.
  • Approved Phase 0 closeout exists.
  • Approved Phase 1 PRD exists.

Exit Criteria

  • The private GitHub repository exists and contains the approved documentation baseline plus the Phase 1 scaffold.
  • The .NET solution and Avalonia application shell exist and run locally on Ubuntu.
  • App-managed local settings and external secret boundaries are implemented and documented.
  • All approved Phase 1 shell views exist in placeholder or inactive-ready form.
  • The build, run, and validation workflow is documented and repeatable.
  • Ignored-file rules and configuration-template boundaries are established.
  • Phase 1 unit, integration, and regression test plans exist.
  • Project context and decision-log artifacts are updated for future sessions.
  • The application can be launched from the documented workflow using checked-in templates plus local settings/secrets conventions without ad hoc setup steps.
  • Editable policy and configuration surfaces behave coherently, persist non-secret operator changes across restart, and communicate invalid or incomplete states clearly.
  • Missing-secret and not-yet-configured states are handled gracefully and visibly in the UI.
  • The approved Phase 0 constraints and Phase 1 product boundaries are visible in the shell rather than existing only in documents.
  • The About view performs a real manual update check against the configured release source without introducing self-updating behavior.

17. Risks and Unknowns

Key Technical Risks

  • Configuration persistence and secret boundaries can become muddled if app-managed settings and template files are not separated cleanly from the start.
  • The shell can over-implement later-phase behavior if placeholder and inactive states are not enforced explicitly.
  • Cross-platform UI concerns can create rework if Ubuntu-first assumptions are not validated early in the Avalonia baseline.
  • Release/update handling can become too coupled to private-repo assumptions if the configured release-source model is ignored.

18. Recommendation

Proceed with Phase 1 implementation using the approved Avalonia/.NET 8 shell baseline, strict external secret handling, app-managed non-secret settings, and a placeholder-first operator UI that preserves later-phase expansion without pretending unimplemented features are active.
