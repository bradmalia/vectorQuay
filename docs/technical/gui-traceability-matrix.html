GUI Traceability Matrix

Mapping Phase 1 GUI shell features to the approved roadmap phases so later implementation scope stays explicit.

Versionv1.0
StatusWorking Baseline
OwnerBrad Malia
ThemeIndigo Signal

Executive Summary

The approved roadmap covers the major functional intent of the Phase 1 GUI shell, but not every PRD mock control is mapped with equal precision. Most core screens are covered explicitly by later phases. Some richer operator-facing controls are only implied by broader roadmap language. Support the Author is now assigned as an optional productization feature under Phase 10 rather than remaining unassigned.

Quick View

Explicit means the roadmap clearly names the function in a later phase. Implicit means the screen feature is likely intended but only through broader roadmap language. Unassigned means the PRD introduces a GUI feature with no clear later roadmap home.

Feature Groups That Need Sharper Assignment

  • Alerts detail: category, severity, channel, and quiet-hour preference granularity is only implied today.
  • Sources detail actions: View History, Automation Settings, reversible source-state workflows, and prompt-driven watcher creation need more precise later-phase ownership.
  • Home as a composed dashboard: the roadmap covers the constituent data, but not the dashboard surface explicitly as a deliverable.
  • Advanced policy UX: bulk actions, deep filtering, and richer selection-state handling should be assigned more explicitly if they are required for completion.

Scope Risk

If the roadmap remains broad while the GUI PRD remains specific, later implementation can drift into “the screen implied it” arguments. This is manageable, but only if later PRDs/TRDs adopt the unresolved PRD controls explicitly.

Recommended Action

  • Use this matrix when drafting Phase 2 and later PRDs/TRDs.
  • Mark each unresolved GUI feature as either explicitly assigned, intentionally deferred, or removed from completion scope.
  • Keep Support the Author optional and productization-only so it does not blur the trading system boundary.
  • For Sources and Alerts, convert broad roadmap intent into specific operator workflows before those phases begin implementation.

Working Conclusion

The major functional shell areas are covered by later phases. The main remaining traceability issue is not missing product direction overall; it is uneven specificity between the GUI PRD and the roadmap. That should be corrected in future phase-definition packages rather than discovered during implementation.
